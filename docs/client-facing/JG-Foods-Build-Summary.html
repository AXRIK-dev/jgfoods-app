Session 4 — Database & wiring

What we built this session

Full Supabase schema — 5 migrations
Every table the system needs is now written and ready to run: products, customers, delivery_slots, orders, order_items, invoices, invoice_items, weekly_sheets, invoice_payments, cash_tab_entries, and delivery_temps. Run them in order in the Supabase SQL editor and the database is live.
Automatic triggers
Three database triggers keep everything consistent without any extra steps from Jon: order totals recalculate automatically when items change, delivery slot counts stay accurate when orders are placed or cancelled, and every table's "last updated" timestamp is maintained automatically.
Row Level Security
The public website can read products and open delivery slots — nothing else. Orders and customer records are admin-only. The public can place an order without ever having read access to the orders table. Account customers can see only their own record and history.
place_order function
Website visitors submit orders through a single secure database function — not by writing directly to any table. The function checks the slot is open, hasn't hit capacity, and the cut-off hasn't passed before creating the customer (or matching an existing one) and inserting the order atomically. Returns a human-readable reference (e.g. JGF-4A2F1C).
Temp log linked to delivery runs
When Jon confirms a delivery slot, the system automatically creates one temperature log entry per customer with an order on that run. Jon opens the Temp Log screen on his phone and sees exactly who he's delivering to that day — no manual list-building, no hunting for the right date. He just taps in the reading as he goes. Previously the temp log used a hardcoded customer list saved to local browser storage — it now reads from and writes to Supabase.
Monthly compliance export ready
A database view (monthly_temp_summary) produces the exact data needed for the monthly printed temperature log Jon files for environmental health — customer name, delivery date, temperature reading, and day label, for any given month.
Supabase client wired into admin app
The admin app now loads the Supabase JS client on startup. One placeholder (the anon key from the Supabase dashboard) needs dropping in — once that's done, every TODO in the code that references Supabase is ready to be activated. The app is connected in structure; the key makes it live.
Go-live checklist created
A full 36-item interactive checklist covering every step needed before the system goes live — Supabase migrations, Netlify deployment, third-party connections (EmailJS, Twilio, Google Maps, Claude API), data loading, pre-launch testing, and handover to Jon. Tickable in the browser with progress saved between sessions.
SQL run order guide
A plain-English guide to the 5 migrations — what each one does, the exact order to run them, and what to do with the anon key afterwards. Saved in the supabase folder alongside the migration files.
Project folder reorganised
The JG Foods project folder is now structured into clear subfolders — admin (app code), website (customer site), supabase (migrations), docs (all documents split into client-facing and internal), assets (logos and graphics), and scripts (portal update tools). Easier to navigate as the project grows.
