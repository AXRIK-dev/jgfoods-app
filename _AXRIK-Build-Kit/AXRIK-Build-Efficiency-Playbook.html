AFTER-ACTION · FIRST BUILD

Doing the next one in half the time.

JG Foods was the first AXRIK build — a full customer website and bespoke back-office for a mobile butcher, delivered in ~35 hours across 7 sessions. This playbook turns everything learned into a reusable kit, a set of skills and a process, so the next niche client starts at ~60% done.

What the first build actually cost

~35BUILD HOURS
7SESSIONS
16DB MIGRATIONS
~8OF THOSE = REWORK

Two apps (1,700-line website + 6,200-line single-file admin), 16 tables, 3 Netlify functions, 3 live AI features. The headline opportunity isn't the features — it's that roughly half the migrations were re-doing work, not adding it. That's the time to claw back.

The five things that caused rework

Each of these is now pre-solved in the kit. This is the core of the time saving.

1. Security written broad, then hardened four times BIGGEST WIN

RLS started as "any logged-in user = full access", then took four later migrations (role-based access, financial-table lockdown, admin hardening, a signup-trigger fix) to get right. Fix: ship role-based RLS — admin / staff / account / public — in the first RLS migration, with financial tables admin-only from day one.

→ Solved in starter-kit/supabase/002_roles_and_rls.sql

2. Trigger edge cases found in production MED

The slot orders_count didn't update when an order was moved between slots (a later fix), and the signup trigger broke on null user metadata (another fix). Both are now baked into the base schema correctly.

→ Solved in starter-kit/supabase/001_base_schema.sql + 002

3. Incremental migrations for one cohesive model MED

16 migrations grew one at a time. For the next build the whole spine — catalogue, customers, slots, orders, items, invoices — lands in three consolidated, customisable files.

4. Config hard-coded, then edited repeatedly MED

Delivery days, cut-offs and copy were coded as constants, so every tweak was a code change. Fix: an app_settings key/value table for anything the client might want to change.

5. Leftover placeholders shipped, then fixed LOW

Stale integration notes (e.g. an unconfigured SMS provider) reached the client and became confusing "fix" commits. Fix: an end-of-session hygiene sweep (prompt #9 in the library).

The kit: what's reusable, and where it lives

Everything below is in _AXRIK-Build-Kit/ in this project folder. Move it to a shared AXRIK location and clone per client.

1 · Starter kit (the ~60% head start)

_AXRIK-Build-Kit/starter-kit/ supabase/001_base_schema.sql core tables + triggers (genericised) supabase/002_roles_and_rls.sql role-based RLS, correct on day one supabase/003_place_order_rpc.sql public ordering, capacity enforced netlify-functions/ai.js generic Claude proxy (all AI features) netlify-functions/manage-users.js client self-manages staff logins README.md 30-min setup + customisation checklist

For any scheduled-delivery / ordering business — mobile trades, farm shops, bakeries, florists, meal-prep — this is the backend before you write new code. Customise the >>>-marked CHECK constraints to the client's reality.

The next-build workflow (target: ~15–18h)

  • Discovery first, schema before screens. Model from the client's real spreadsheets/messages — not assumptions. The two ends are one system sharing one database.
  • Day one: clone the starter kit, run 001→002→003, customise the markers. Backend secure and ordering-ready in ~30 min.
  • Website → admin → AI → go-live, in that order. AI layers on last, always with a fallback.
  • Build config into app_settings, not constants.
  • End every session with the hygiene sweep (prompt #9): every table has RLS, every AI button has a fallback, no stale placeholders, stubs listed.
  • Handover from templates: go-live checklist, client-runnable testing worksheet, build summary, "by the numbers".

What to decide next

  • Where the kit lives. Move _AXRIK-Build-Kit/ into the shared AXRIK folder so it's not buried inside one client.
  • Install the four skills via Settings → Capabilities so they trigger automatically on the next project.
  • After 2–3 more builds, revisit the SaaS-vs-managed-template question — by then the starter kit will tell you how much is genuinely shared across niches.
